# RESEARCH.md — AIエージェント設計へのQA統合：現状調査

作成日: 2026-06-01

## 調査の動機

AIエージェントが設計や計画を行う際、QA（品質保証）の観点の導入が重要であるという仮説のもと、既存研究・実装を調査した。

---

## Part 1: 学術調査結果（deep-research ワークフロー）

107エージェント・234万トークンを使用した大規模調査。

### 検証済みクレーム（信頼できる知見）

| 知見 | 信頼度 | 投票 | 主要ソース |
|------|--------|------|-----------|
| リフレクション・洗練メカニズムはフォールトトレランスを向上させるが、収束の形式的保証がない | High | 3-0 | arXiv:2402.02716, arXiv:2310.01798 |
| タスク特化型サブエージェント分解（TDFlowパターン）はQA的役割分担の実装パターンとして機能する | Medium | 2-1 | arXiv:2510.23761 |

### 棄却されたクレーム（根拠が弱い主張）

| 主張 | 棄却票 | 棄却理由 |
|------|--------|---------|
| 自動品質ゲート（PROMOTE/HOLD/ROLLBACK）フレームワークの有効性 | 1-2 | 再現性・独立検証不足 |
| TDD/BDDをAIエージェントにそのまま適用する効果 | 0-3 / 1-2 | モデル依存性が高く一般化不可 |
| 内因的自己批判のみで計画品質を向上できるという主張 | 0-3 | 外部フィードバックなしの限界が複数ソースで確認 |

### TDFlowパターン（最も有望な実装パターン）

SWE-Bench Lite 88.8%（human-written tests使用時）を達成した4サブエージェント分解：

```
Explore Files → Debug One → Revise Patch → Generate Tests
     ↑             ↑              ↑               ↑
  調査特化     診断特化       修正特化        テスト特化
```

**重要な制限**: LLM生成テストでは68.0%に低下（human-written: 94.3%）。
テスト生成品質自体が未解決課題。

### 未解決の重要問題

1. **収束保証問題** — 外部検証器をエージェント計画ループに統合した場合、収束は理論的に保証できるか？
2. **ドメイン汎化問題** — TDFlowパターンをコード生成以外（要件分析、アーキテクチャ設計）に適用できるか？
3. **AIエージェント固有のQA指標** — 非決定的出力・確率的実行に対して何を計測すべきか？
4. **無限後退問題** — 品質ゲートキーパーエージェント自体の信頼性をどう保証するか？

## Part 2: 要件工学とQA観点の分析

### Zenn記事「Requirements Engineering」の主要知見

出典: https://zenn.dev/rehabforjapan/articles/requirements-engineering

#### 核心的な主張：「操作」ではなく「状態」から思考する

> 画面操作から要件を考え始めるとハッピーパスのみになる

**推奨アプローチ**:
1. 全データ状態を先に定義（エラー状態・権限不足・データなし等）
2. 操作は後付けで導出する（状態と操作結果との整合性を確認するフェーズがあると良さそう）

#### 3段階要件工学プロセス

| フェーズ | 内容 |
|---------|------|
| 概念整理 | 定義・責務・範囲・方針を構造化 |
| 合意形成 | 関係者による承認 |
| 仕様生成 | 具体的な仕様書ドラフト作成 |

#### 「思想」（幹）と「仕様」（枝葉）の区別

| 観点 | 思想（概念整理） | 仕様（仕様生成） |
|------|-----------------|-----------------|
| 性質 | 判断基準・理想像 | 具体的実装形式 |
| 変化頻度 | フェーズ跨ぎで不変 | フェーズ/優先度で更新 |
| 姿勢 | 丁寧に構築 | 変更前提で割り切る |

## Part 3: make-plan.mdの操作中心バイアス分析

### 問題の構造

`~/.claude/commands/make-plan.md` の現状：

| セクション | 何を検証しているか | 問題 |
|-----------|-----------------|------|
| HTTP Status Coverage（L506-L516） | 操作の結果（HTTPレスポンスコード） | 操作中心 |
| Red/Green/Refactor（L527-L553） | 操作の手順（テスト→実装→改善） | 操作中心 |
| Manual Verification（L557-L559） | 操作と期待挙動 | 操作中心 |
| Behavior Spec → `io_mapping`（L209-L210） | 入出力マッピング | 操作中心 |

**構造的欠陥**: `state_transitions`（データ状態遷移）は `reactive` 型のみ必須。
`transformation` 型（API実装・関数）では**データ状態分析がスキップされる**。

### 欠落している状態中心アサーション

```yaml
# 現在のmake-plan.mdが生成するもの（操作中心）
io_mapping:
  - input: POST /api/items {name: "foo"}
    output: 200 {id: "123", name: "foo"}

# 欠落しているもの（状態中心）
pre_conditions:
  - items テーブルに同名レコードが存在しない
post_conditions:
  - items テーブルに {name: "foo", status: "active"} のレコードが1件増加する
invariants:
  - items.created_at は常に items.updated_at 以下
  - 削除済みレコードは name の一意制約から除外されない  # エラー状態
```

### 学術研究との接続

本日の deep-research で棄却された「TDD直接適用の有効性（0-3票）」と根が同じ問題：
- TDDのRed-Green-Refactorは「操作が成功するか」を確認する
- 「操作後にデータが正しい状態にあるか」は別の問題
- 外部フィードバックシグナル（データ状態の実際の確認）なしには品質保証できない

## Part 4: 改善仮説

### make-plan.mdへの追加候補

1. `pre_conditions` / `post_conditions` を `io_mapping` の必須フィールドに格上げ（DbC形式）
2. `data_invariants`（不変条件）セクションを Behavior Spec に追加
3. Manual Verificationにデータ状態の確認手順を明示的に要求
4. `transformation` 型システムでも状態遷移の部分的な記述を義務化

### 優先度評価

| 改善案 | 効果 | 実装コスト | 推奨度 |
|--------|------|-----------|-------|
| `pre_conditions`/`post_conditions` 必須化 | 高 | 低 | ⭐⭐⭐⭐⭐ |
| `data_invariants` セクション追加 | 高 | 中 | ⭐⭐⭐⭐ |
| Manual Verificationへのデータ状態確認 | 中 | 低 | ⭐⭐⭐⭐ |
| transformation型への state_transitions 部分適用 | 中 | 高 | ⭐⭐⭐ |

## 結論

AIエージェントの設計・計画フェーズへのQA統合において：

1. **外部フィードバック信号が必須**（自己批判のみでは不十分、3-0票で確認）
2. **TDDの直接適用は効果不十分**（操作中心バイアスが残る）
3. **状態中心の検証アプローチ**が理論・実践の両面から支持される
4. **make-plan.mdは操作中心バイアスを持っており**、状態アサーションの追加が急務
