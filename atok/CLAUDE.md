# ATOK35 ローマ字コンバーター - プロジェクト知見

## 概要

`atok_convert.py` は ATOK35 のテキストエクスポート (`default(ローマ字出力).txt`) から StUser0.plist のローマ字バイナリを再生成するコンバーター。

## ファイル配置

| ファイル | パス |
|---------|------|
| コンバーター | `~/repos/dotfiles/atok/atok_convert.py` |
| 入力テキスト | `~/Library/CloudStorage/Dropbox/default(ローマ字出力).txt` (UTF-16 BE) |
| 出力 plist | `~/Library/Preferences/ATOK35/Styles/StUser0.plist` |
| バックアップ | 同上 `.bak` (コンバーター実行時に自動生成) |

## バイナリフォーマット (tag_RomaKanaInfo, Big-Endian, 16392 bytes)

| セクション | オフセット | サイズ | 内容 |
|-----------|----------|--------|------|
| Header | 0 | 8 | uint32 entry_count + uint32 magic (0x01000002) |
| Section A | 8 | 384 | 96ペア × (uint16 start, uint16 count) - 文字コード(32-127)のルックアップ |
| Section B | 392 | 4000 | 1000エントリ × (uint16 c_offset, uint8 in_len, uint8 out_len) |
| Section C | 4392 | 12000 | 6000 uint16 - 入力/出力コードの実データ |

## 過去のバグと教訓

### バグ1: テキストエクスポートに記号ルールが含まれない (2026-04-04 修正)

**症状**: `,` `.` `[` `]` キーで句読点・括弧が入力されない
**原因**: ATOKのテキストエクスポートは記号→句読点ルール (`,`→`、`, `.`→`。`, `[`→`「`, `]`→`」`) を出力しない。これらはATOK内部の暗黙デフォルトだが、バイナリには明示的に必要。
**修正**: `main()` でデフォルト記号ルールを自動追加。`KANA_TO_HW` テーブルに `、。「」` の半角変換を追加。

### バグ2: Section A 未使用エントリの start=0 問題 (2026-04-04 修正)

**症状**: `-` キーが `。` を出力、`@` キーが `ー` を出力 (キーマッピングのずれ)
**原因**: Section A の未使用エントリ (count=0) の start 値が 0 のまま初期化されていた。ATOKのルックアップエンジンが未使用エントリの start 値を範囲計算に使用するため、start=0 だと B[0] のルールを誤参照する。
**修正**: forward-fill 方式に変更。未使用エントリの start は「直前の有効エントリの終端インデックス」を指すようにした。
**検証方法**: Section A の全 start 値が単調非減少であること。

### plist 書き込み時の注意 (既知)

ATOK35 の独自パーサーは **1行 base64** しか読めない。`plistlib.dump()` は改行付き base64 を出力するため使用不可。テキストレベルの regex 置換で `<data>` タグ内の base64 を差し替える必要がある。

## テスト手順

1. `python3 atok_convert.py` を実行
2. 出力の検証セクションで先頭エントリを確認:
   - `B[0]` が `,` → `､` (読点)
   - `B[1]` が `-` → `ｰ` (長音)
   - `B[2]` が `.` → `｡` (句点)
3. ATOKを再起動して動作確認
4. 問題があれば `.bak` から復元可能

## カスタムルール

テキストファイルにはユーザー定義のカスタムルールが含まれる:
- `;n` → `なかった`
- `@n` → `ならない`
- `n'` → `ん` (明示的なN確定)

これらは正常な動作であり、コンバーターのバグではない。
